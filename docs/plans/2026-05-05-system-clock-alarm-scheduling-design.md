# 系统时钟闹钟调度设计

日期：2026-05-05
状态：按用户最新约束确认

## 目标

- 所有课程闹钟必须通过系统时钟 App 设置，不使用 App 自己响铃作为正式闹钟方案。
- 用户刚创建提醒规则时，如果规则对应当天尚未过去的触发时间，则立即创建系统时钟闹钟。
- 每天 22:00 检查并创建下一天需要的系统时钟闹钟。
- 每节课下课时检查并创建后续需要的系统时钟闹钟。
- 添加闹钟前先判断 App 是否已经登记过同一个闹钟，避免重复调用系统时钟 App。
- 只有系统时钟闹钟创建调用成功后，才登记为已添加。

## App 与桌面小组件协同

App 本体和桌面小组件都调用 `ReminderCoordinator`，共用 `ReminderRepository` 中的规则和系统闹钟登记簿。`ReminderCoordinator` 内部用进程级互斥锁包住“读取登记簿、删除失效闹钟、提交新闹钟、写回登记簿”的完整事务，避免 App 和小组件同时触发时重复设置同一个系统闹钟。

桌面小组件不是独立常驻进程，但它是桌面可见的 App 组件：桌面刷新、小组件更新、用户点击和周期刷新 worker 都能唤起同一套同步逻辑。小组件侧不维护第二份规则或闹钟状态，只读取 App 的 DataStore 数据并调用同一协调器。

## 明确不做

- 不做响铃时长设置。
- 不做响铃间隔设置。
- 不做响铃次数设置。
- 不用多条闹钟模拟间隔或次数。
- 不把应用内闹钟、通知或后台任务当成正式闹钟回退方案。

## Android 官方接口边界

系统时钟 App 的标准入口是 `AlarmClock.ACTION_SET_ALARM`。该接口能传入小时、分钟、消息、铃声、震动、是否跳过 UI 等标准字段，但没有标准字段可以设置单个闹钟的响铃时长、响铃间隔或响铃次数。

Android 没有标准 API 读取系统时钟 App 中已经存在的闹钟列表，因此 App 无法可靠确认用户是否在系统时钟 App 里手动删除了某个闹钟。去重只能基于 App 自己记录的“本 App 曾经成功创建过”的登记簿。

Android 提供 `AlarmClock.ACTION_DISMISS_ALARM` 用来请求系统时钟按搜索条件关闭闹钟。为避免误删用户自己创建的闹钟，本 App 创建系统闹钟时使用带短哈希的唯一标签，并在下课检查、规则删除、课表变化或小组件刷新时只按这个唯一标签请求删除本 App 记录过的闹钟。

`ACTION_SET_ALARM` 也不能指定一次性闹钟的具体日期，只能表达“下一次这个时分”。因此代码会拒绝提交会被系统时钟误解到错误日期的计划。例如 22:00 检查明天闹钟时，明天 22:30 的闹钟不能在今天 22:00 用标准接口安全创建，否则系统时钟可能把它建成今天 22:30；这类计划会等到当天后续下课检查或当天创建规则时再处理。

参考：

- Android `AlarmClock` API: https://developer.android.com/reference/android/provider/AlarmClock
- Android 闹钟调度说明: https://developer.android.com/develop/background-work/services/alarms/schedule

## 核心方案

保留现有提醒规则模型和 `ReminderPlanner`，新增系统时钟闹钟登记簿。调度流程分成“规划”和“提交”两步：

1. `ReminderPlanner` 根据规则、课表、时间表和临时调课生成 `ReminderPlan`。
2. 新增调度窗口筛选逻辑，只取本次检查窗口内的计划。
3. 提交前查询闹钟登记簿，已登记成功的计划不重复提交。
4. 未登记的计划通过 `SystemAlarmClockDispatcher` 调用 `AlarmClock.ACTION_SET_ALARM`。
5. 只有 `startActivity()` 成功返回，才把该计划写入登记簿。
6. 如果系统时钟 App 不存在、拒绝权限、厂商实现异常或启动失败，只记录失败结果，不写登记簿。

## 检查时机

### 创建规则时

创建规则后只检查当天：

- 窗口开始：当前时间。
- 窗口结束：当天 23:59:59。
- 若规则当天有尚未过去的触发时间，则立即设置对应系统时钟闹钟。
- 若触发时间已过去，不补设今天的闹钟。

### 每天 22:00

由 `SystemAlarmCheckReceiver` 接收一个内部精确闹钟：

- 检查窗口：明天 00:00:00 到明天 23:59:59。
- 为明天所有符合规则、未登记成功且能被 `ACTION_SET_ALARM` 安全表达为下一次时分的计划创建系统时钟闹钟。
- 完成后继续安排下一次 22:00 检查。

这个内部检查闹钟只用于唤起检查逻辑，不替代系统时钟 App 闹钟。

### 每节课下课时

为当天课程下课时间安排内部检查：

- 检查窗口：检查触发时刻到当天 23:59:59。
- 用于补齐当天后续未登记成功的系统时钟闹钟。
- 每次完成检查后，安排当天下一次下课检查；没有下一节课时等待 22:00 每日检查。

这个内部检查闹钟同样只负责执行检查，不作为课程闹钟。

## 闹钟登记簿

新增 `SystemAlarmRecord` 数据模型，存储在 DataStore：

- `alarmKey`：稳定去重键。
- `ruleId`：提醒规则 ID。
- `planId`：提醒计划 ID。
- `pluginId`：插件 ID。
- `triggerAtMillis`：课程提醒触发时间。
- `createdAtMillis`：系统时钟创建调用成功时间。
- `message`：写入系统时钟 App 的闹钟消息。
- `alarmLabel`：写入系统时钟 App、用于后续 `ACTION_DISMISS_ALARM` 标签搜索的唯一标签。

`alarmKey` 使用 `pluginId + triggerAtMillis + title + message + ringtoneUri` 生成，让同一规则或同定义重复规则不会重复创建同一条系统闹钟。登记簿只代表“本 App 曾成功请求系统时钟创建该闹钟”，不代表该闹钟此刻仍存在于系统时钟 App。

清理策略：

- 删除提醒规则时，先按登记标签请求系统时钟删除，再删除该规则对应登记记录。
- 清空课表或清空全部数据时，先按登记标签请求系统时钟删除，再删除所有登记记录。
- 每次同步前只清理早于当前时间的登记记录，不能用“窗口开始时间”清理，避免 22:00 检查明天闹钟时误删今天晚间尚未触发的闹钟。
- 同步窗口内如果存在登记簿记录但当前课表/规则已经不再生成对应计划，则视为失效闹钟，按标签请求删除。

## 成功与失败判断

成功标准：

- `SystemAlarmClockDispatcher.dispatch(plan)` 调用 `context.startActivity(intent)` 没有抛出异常。
- 满足该标准后返回成功结果，并写入登记簿。

失败标准：

- `ActivityNotFoundException`：没有可处理系统时钟入口。
- `SecurityException`：系统或厂商实现拒绝设置闹钟。
- 其它 `RuntimeException`：厂商时钟实现或系统行为异常。

失败时不写登记簿，下次检查仍会再次尝试。

## 防崩溃

- 所有系统时钟调用都捕获异常并返回结构化结果。
- BroadcastReceiver 只启动协程/检查工作，不在主线程做长耗时操作。
- 检查逻辑集中在 coordinator，UI、Receiver 和应用启动只调用统一入口。
- DataStore 读写失败只记录日志并返回失败结果，不阻断 App 主流程。
- 计划窗口为空时直接返回，不调用系统时钟 App。
